# OSSAF Local - 安全规范

> 本文档定义 OSSAF Local 项目的凭证隔离、提交防护、漏洞披露与密钥轮换规范。所有贡献者必须遵守。

---

## 一、三层凭证隔离规范

OSSAF Local 采用**三层凭证隔离模型**，确保真实密钥永不入仓：

| 层级 | 文件 | 用途 | Git 状态 | 可包含真实密钥 |
|------|------|------|----------|----------------|
| L1 模板层 | `.env.example` | 团队共享配置占位符 | ✅ 入仓 | ❌ 禁止 |
| L2 Mock 层 | `.env.mock` | 公开 Mock 凭证，本地开发与 CI | ✅ 入仓 | ❌ 禁止 |
| L3 本机层 | `.env.local` | 本机真实密钥，仅本机生效 | ❌ gitignore | ✅ 允许 |

### 凭证流转规则

1. **新增配置项**：先在 `.env.example` 添加占位符 → 再同步至 `.env.mock` 与 `.env.local`
2. **获取真实密钥**：仅写入 `.env.local`，禁止修改 `.env.example` 或 `.env.mock`
3. **密钥轮换**：仅替换 `.env.local` 中的对应字段，无需改动入仓文件
4. **direnv 自动加载**：进入目录自动加载 `.env.local`，离开目录自动 unset 环境变量

### 占位符约定

- 模板层占位符：`SK-PLACEHOLDER-DO-NOT-USE`
- Mock 层公开凭证：`SK-MOCK-{SERVICE}-DEV-001`
- 真实密钥格式：由各服务方提供，禁止与上述两种格式冲突

## 二、禁止 `--no-verify` 硬规则

### 硬规则

```bash
# ❌ 严禁使用以下命令绕过 Git Hooks
git commit --no-verify
git push --no-verify

# ✅ 正确做法：修复 hook 报错后再提交
```

### 执行机制

1. **pre-commit hook**：自动扫描暂存区，检测真实密钥特征（高熵字符串、已知密钥前缀）
2. **commit-msg hook**：校验提交信息规范（Conventional Commits）
3. **pre-push hook**：执行完整测试套件，任一失败则阻止推送
4. **CI 门槛**：即使本地绕过，CI 流水线仍会拦截

### 违规处理

- 首次违规：警告并要求回滚提交
- 再次违规：撤销直接推送权限，改为 PR 审核制
- 造成密钥泄露：立即启动密钥轮换预案（见第四节）

## 三、漏洞披露流程

### 报告渠道

- **安全邮箱**：security@ossaf.local（示例，需替换为真实邮箱）
- **私有渠道**：禁止在公开 Issue 中披露安全漏洞
- **响应时限**：收到报告后 48 小时内确认，7 个工作日内给出修复方案

### 披露流程

1. **报告方**：通过安全邮箱私下报告漏洞详情
2. **维护方**：确认漏洞并评估严重程度（CVSS 评分）
3. **修复期**：
   - 严重漏洞（CVSS ≥ 9.0）：24 小时内修复
   - 高危漏洞（CVSS 7.0-8.9）：72 小时内修复
   - 中危漏洞（CVSS 4.0-6.9）：7 天内修复
   - 低危漏洞（CVSS < 4.0）：下个版本修复
4. **披露期**：修复发布后 90 天，公开漏洞详情与修复方案

### 报告模板

```
漏洞标题：
严重程度：[严重/高/中/低]
影响版本：
复现步骤：
1.
2.
3.
预期行为：
实际行为：
修复建议：
报告人联系方式：
```

## 四、密钥轮换预案

### 触发条件

- 密钥确认泄露（误提交、日志输出、第三方泄露）
- 定期轮换周期到达（建议每 90 天）
- 团队成员离职（针对个人令牌）
- 服务方强制要求轮换

### 轮换步骤

#### 步骤 1：隔离泄露源（5 分钟内）
```bash
# 立即在密钥所属服务的控制台禁用旧密钥
# 确认 .env.local 中的旧密钥已被服务方拒绝
```

#### 步骤 2：生成新密钥（10 分钟内）
```bash
# 在各服务控制台生成新密钥：
# - Anthropic Console: https://console.anthropic.com/
# - OpenAI Platform: https://platform.openai.com/api-keys
# - Google Cloud Console: https://console.cloud.google.com/
# - GitHub Settings: https://github.com/settings/tokens
# - AlternativeTo API: 联系服务方
```

#### 步骤 3：更新本机配置（5 分钟内）
```bash
# 仅修改 .env.local，禁止修改入仓文件
vim .env.local
# 替换对应字段为新密钥

# 重启 Mock 服务器使新密钥生效
make mock-stop
make mock-start
```

#### 步骤 4：清理历史泄露（如适用）
```bash
# 若密钥已误提交到 Git 历史：
# 1. 立即禁用旧密钥（步骤1已完成）
# 2. 使用 git filter-repo 清理历史（谨慎操作）
# 3. 强制推送前需团队全员确认
# 4. 通知所有 Fork 仓库同步清理
```

#### 步骤 5：记录与复盘
- 在内部文档记录泄露事件、影响范围、修复措施
- 复盘根因，完善防护机制
- 必要时对外公告

## 五、Mock 数据合规要求

### Mock 数据边界

1. **仅限本地开发**：Mock 凭证 `SK-MOCK-*` 仅在本地开发环境与 CI 环境使用
2. **无真实访问能力**：Mock 凭证不提供任何真实 API 访问能力，仅触发 Mock 响应
3. **可公开提交**：`.env.mock` 可安全入仓，不构成泄露风险
4. **禁止混用**：生产环境严禁加载 `.env.mock`，开发环境严禁加载生产配置

### Mock 服务器规范

1. **绑定地址**：仅绑定 `127.0.0.1`，禁止监听公网或局域网
2. **端口隔离**：使用 `8765` 端口，避免与生产服务冲突
3. **日志脱敏**：Mock 服务器日志禁止输出任何请求头中的凭证字段
4. **响应合规**：Mock 响应数据使用合成数据，禁止包含真实用户信息

### Mock 数据生成原则

- 使用 Faker 等库生成合成数据
- 禁止复制真实 API 响应数据到 Mock
- Mock 数据集变更需经过 Code Review

## 六、禁止事项（10 条硬规则）

> 以下 10 条为硬规则，违反任何一条均视为严重违规。

### 1. ❌ 禁止将真实密钥写入入仓文件
真实密钥仅允许写入 `.env.local`（已被 gitignore）。禁止写入 `.env.example`、`.env.mock`、`.envrc`、`Makefile`、`README.md` 或任何代码文件。

### 2. ❌ 禁止使用 `git commit --no-verify` 绕过 Hooks
所有提交必须通过 pre-commit、commit-msg、pre-push hook 校验。Hook 报错必须修复，不允许绕过。

### 3. ❌ 禁止在日志中输出凭证
任何日志输出（console、文件、stdout/stderr）禁止包含 `*_API_KEY`、`*_TOKEN` 等凭证字段的明文值。日志中凭证字段必须脱敏为 `SK-***`。

### 4. ❌ 禁止将 Mock 服务器暴露至公网
Mock 服务器必须绑定 `127.0.0.1`，禁止绑定 `0.0.0.0` 或公网 IP。禁止通过 ngrok、frp 等工具将 Mock 服务器映射至公网。

### 5. ❌ 禁止在 Git 历史中保留泄露密钥
一旦发现密钥误提交，必须立即禁用该密钥并清理 Git 历史（使用 `git filter-repo`）。禁止仅删除最新提交而保留历史中的泄露记录。

### 6. ❌ 禁止共享 `.env.local` 文件
`.env.local` 为本机私有文件，禁止通过任何渠道共享（邮件、聊天、云盘、U 盘）。团队新成员应自行从服务方获取密钥并填写本机 `.env.local`。

### 7. ❌ 禁止在测试用例中硬编码真实密钥
测试用例必须使用 Mock 凭证或环境变量读取。禁止在 `tests/` 目录下任何文件中出现真实密钥字符串。

### 8. ❌ 禁止禁用 SSL/TLS 证书校验
所有 HTTP 客户端必须启用证书校验。禁止设置 `verify=False`、`NODE_TLS_REJECT_UNAUTHORIZED=0` 或忽略证书错误。

### 9. ❌ 禁止将生产环境数据导入本地开发
本地开发仅允许使用 Mock 数据或脱敏后的合成数据。禁止从生产数据库导出真实数据至本地环境。

### 10. ❌ 禁止在 CI 中使用个人令牌
CI 环境必须使用服务方提供的 CI 专用令牌（如 GitHub Actions 的 `GITHUB_TOKEN`、Deploy Key 等），禁止使用团队成员的个人令牌。CI 令牌必须具备最小权限原则。
